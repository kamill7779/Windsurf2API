# 原生桥接模式 — Bash 回灌

**两种可切换的 Anthropic 工具方法之一。** 选型说明见 [`README-zh.md`](./README-zh.md)。

这个方法通过复用 Windsurf Cascade 原生已暴露的 `run_command` 提案链路，来支持 Claude Code 风格的 `Bash` 工具调用。它把一次 Cascade 原生提案对外转成 Anthropic `tool_use`，让下游客户端本地执行命令，然后把执行结果回灌到同一条 Cascade 会话中，让生成过程继续向前推进。

## 短版本

1. 下游通过 Anthropic `tools[]` 请求 Bash 工具。
2. 上游 Windsurf 原生规划产出 `run_command` 提案。
3. 代理把提案映射为 Anthropic `tool_use(name="Bash")`。
4. 客户端本地执行并返回 `tool_result`，携带 stdout、stderr、退出码。
5. 代理把结果回灌到同一条 Cascade 轨迹，让模型继续生成。

这 **不是** 提示词注入，也 **不是** 把 Bash 交给语言服务器直接执行。它的本质是：

- 原生工具提案捕获
- 客户端本地执行
- 结果回灌到原始会话
- 原轨迹继续生成

## 适用范围

适用于：

- `POST /v1/messages`
- Claude Code 风格的 Bash 工具调用
- 命令必须在客户端本地环境执行的场景

不覆盖：

- 非 Bash 的任意工具桥接
- OpenAI / Responses 接口的工具通道（这两个端点在该模式下工具仍不支持）
- 必须运行在上游模型自身沙箱中的工具流程

## 核心思路

把一次工具轮次拆成两段。

**上半段 — 由 Windsurf 完成：**

- 接收对话
- 在原生规划阶段，提出 `run_command` 和具体命令行

**下半段 — 由代理和客户端共同完成：**

- 代理把提案转成 Anthropic `tool_use(Bash)`
- 客户端在本地环境中执行
- 代理把结果回灌到同一条 Cascade 轨迹

对下游看起来这是标准 Anthropic 工具轮次；对上游看起来这仍然是同一条原生 Bash 交互链路的延续。

## 端到端流程

### 1. 接收带 Bash 工具的 Anthropic 请求

代理检查 `tools[]`，识别是否包含可映射为 Bash 的工具声明（例如名为 `Bash`、`bash` 或 `run_command` 的工具）。

如果没有可映射工具，则退出本模式，请求按普通 Anthropic 文本请求处理或按不支持拒绝。

### 2. 启动原生规划并捕获提案

代理以默认规划模式发起普通 Cascade 请求，允许 Windsurf 走自己的原生工具规划。

在流式轨迹中一旦出现 `run_command` 提案，代理就把它视为一次可外放的 `tool_use` 候选，并同时记录恢复该轨迹所需的最小上下文：

- cascade 标识
- 轨迹标识
- 提案所在步骤
- 对应的工具调用标识
- 提案命令行

### 3. 对外返回 Anthropic `tool_use`

捕获提案之后，代理不会让上游继续沿内部执行路径走完，而是立刻向下游返回一个 Anthropic 兼容响应：

- `tool_use` 块，`name: "Bash"`，输入为提案命令
- `stop_reason: "tool_use"`

下游客户端由此接管执行。

### 4. 接收客户端的 `tool_result`

客户端本地执行完毕后返回 `tool_result`，代理从中提取：

- `tool_use_id`
- stdout
- stderr
- 退出码

`tool_use_id` 被用来找回之前保存的桥接会话，确保结果被送回正确的原始轨迹。

### 5. 通过交互通道回灌（默认路径）

默认回灌策略不是"伪造一段普通文本"，而是沿用 Windsurf 已有的 `run_command` 交互确认通道。

代理会把客户端执行结果合成为一条能够复现该结果的 shell 回放命令：

- 如果命令成功且无错误输出，回放命令仅复现 stdout
- 如果存在 stderr 或非零退出码，回放命令同时复现 stdout、stderr 和原始退出码

随后代理以"确认执行"的方式，把这条回放命令提交给原提案所在的步骤。目标 **不是** 再次真实执行命令，而是让上游状态机接受"这次 Bash 工具已经产生了这些结果"。

### 6. 收集续跑并返回最终回答

回灌被接受后，代理继续拉取原始 Cascade 轨迹的后续步骤，再把产出的 assistant 文本包装为标准 Anthropic assistant 响应返回给客户端。

从客户端视角，这构成完整闭环：

- `tool_use`
- 本地执行
- `tool_result`
- 最终 `assistant` 回答

## 降级路径

如果交互通道没能让轨迹继续推进，代理会降级到更保守的两段路径：

1. **让 cascade 回到空闲态。** 取消当前 invocation 并等待其回到 idle。
2. **回放路径 A — 追加步骤。** 以新用户消息携带合成的 `run_command` 与命令状态步骤，请求上游从该处继续。
3. **回放路径 B — ground-truth 重放。** 如果追加步骤仍无法推进，则把原提案步骤连同合成的结果步骤一起，通过 ground-truth 重放通道一次性提交。

因此整个桥接呈现为 **交互优先，回放兜底** 的双层结构。

## 为什么需要合成 shell 回放

下游返回的是结构化执行数据；而上游已知稳定的入口是 Bash 交互确认。两者不是同一种数据形态。

合成的回放命令充当 **语义转换器**：

- 不要求语言服务器理解 `tool_result` 结构
- 不要求再次真实执行原始命令
- 只需上游状态机接受一份等价的命令执行结果

这一步是整条回灌路线能够成立的基础。

## 优势

- 保持 Anthropic 外部语义，Claude Code 可按标准工具协议工作
- 不依赖提示词模拟，模型看不到自定义工具协议
- 复用已验证的 Windsurf 原生交互通道
- Bash 在客户端本地执行，符合工具本地使用方式
- 双层恢复：交互优先，回放兜底

## 限制

- 当前仅覆盖最小 Bash 桥接；其他工具提案不会自动走这条路径
- 依赖上游实际产出 `run_command` 提案；若原生规划选择了别的路径，本模式不会对外产生工具调用
- 假设客户端能够可靠地在本地执行 Bash 并返回结构化 stdout / stderr / 退出码

## 验证清单

至少应验证：

1. 带 Bash 工具声明的请求返回标准 Anthropic `tool_use`。
2. `tool_use` 中携带的命令与上游原生提案一致。
3. 后续 `tool_result` 能够依据 `tool_use_id` 命中正确的桥接会话。
4. 回灌后原始 Cascade 轨迹能够继续推进，而不是停在提案步骤。
5. 最终返回给客户端的是普通 assistant 消息，不泄露内部桥接痕迹。
6. 当交互回灌失败时，降级路径要么完成，要么给出明确的失败边界。

## 何时选择此模式

以下场景应选择 `native_bridge`：

- 目标是先打通 Claude Code 风格的 Bash 工具闭环
- 希望命令在客户端本地环境执行
- 希望尽量复用 Windsurf 已验证的原生交互链路
- 明确不希望向模型提示词中注入工具协议

若需要覆盖 Bash 以外的任意工具，并能接受提示词级协议，优先选择 [`prompt_emulation`](./system-prompt-tool-emulation-zh.md)。
